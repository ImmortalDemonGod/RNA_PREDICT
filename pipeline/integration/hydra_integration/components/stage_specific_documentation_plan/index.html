    HYDRA & STAGE-SPECIFIC DOCUMENTATION: MASTER PLAN

    Below is a unified, in-depth reference that merges your existing Hydra Integration Plan with a systematic approach to generating specialized documentation for each pipeline stage (AD) and the Unified Latent Merger. It aims to ensure that when you refactor each stage, you'll have thorough reference materials connecting code logic, Hydra configuration, algorithmic details, and integration points.

    1. Comprehensive Feature Documentation (Hydra) Already Done

    You already have a "Hydra Integration Master Document" that: - Explains why Hydra is beneficial (centralization, easy overrides, reproducibility) - Details how to structure conf/ with YAML files, typed dataclasses in config_schema.py, and how to refactor each stage to read cfg instead of inline defaults - Includes acceptance criteria ensuring backward compatibility, successful CLI overrides, and default param matching

    That comprehensive Hydra document covers the feature's high-level design, scope, acceptance criteria, step-by-step implementation plan, and references to all pipeline stages. It's your central guide for implementing Hydra across the entire codebase.

    2. General Considerations (Hydra + Documentation)

    You've also enumerated general considerations such as:

    • Code directory structure in rna_predict/conf/
    • Logging HPC environment or partial checkpoint loading
    • The need for iterative refinement of docstrings and minimal HPC notes, etc.

    These ensure your Hydra-based approach remains robust over time.

    3. Plan to Generate Specialized Documentation Per Stage

    Now, we focus on task-specific documentation for each stage and the unified latent merger. The goal is to create a dedicated reference for each major component in your pipeline, clarifying how Hydra-managed parameters align with the stage's logic and function calls.

    3.1 Overall Methodology

    1. Automated Extraction
    2. Use a doc generator (Sphinx, PyDoctor, or similar) to produce base-level references from docstrings, class definitions, and module docblocks

    3. This yields an initial skeleton capturing function/class signatures, parameters, and any existing docstring descriptions

    4. Manual Enrichment

    5. Conduct a structured code review for each stage. Identify:
      • "Magic numbers," hidden assumptions, or specialized data transformations
      • Key integration points (e.g., adjacency from Stage A feeding into Pairformer in Stage B)
      • Known edge cases or fallback logic (like Stage C legacy reconstruction)

    6. Merge these findings into your doc generator output to create in-depth references that go beyond auto-generated docs

    7. Documentation Outline
      For each stage (AD plus the Unified Latent Merger), follow a consistent template:

    8. Purpose & Overview
      • Short paragraph explaining the stage's function and motivation (e.g., "Stage A: Generate a contact matrix for RNA base pairs.")
    9. Inputs & Outputs
      • Data shapes, types, and any assumptions (sin/cos angles, adjacency dimensions, etc.)
      • Expected Hydra config parameters that might affect these shapes or the stage's logic
    10. Key Classes & Methods
      • Summaries of the top-level classes and methods (like StageARFoldPredictor) with bullet-point commentary about their roles
      • Focus on lines or subroutines that are vital for understanding how Hydra's config drives them
    11. Configuration Parameters
      • A listing of the YAML fields relevant to this stage (e.g., stageA.num_hidden, stageA.dropout, etc.), including defaults, usage, and any synergy with upstream or downstream parameters
      • Example of how these values appear in the YAML (conf/model/stageA.yaml) and how they map to the code
    12. Integration & Data Flow
      • Which module feeds into this stage, and which stage(s) rely on its outputs?
      • If adjacency from Stage A is used by TorsionBERT or Pairformer, note that explicitly
    13. Edge Cases & Error Handling
      • Mention what the code does when inputs are malformed, config is out-of-range, or external files (e.g., checkpoint paths) are missing

    14. References & Dependencies

      • Links to external docs (like the RFold paper) or any specialized library usage (like LoRA injection modules, MP-NeRF references)

    15. Iterate & Validate

    16. Circulate these specialized docs among developers for correctness
    17. Update them in tandem with code changes or as part of the PR (pull request) process to keep them fresh

    4. Stage-by-Stage Documentation Breakdown

    This section sketches an ideal, final content structure for each specialized doc. You can store them either in separate Markdown files (one per stage) or unify them under one "RNA_PREDICT Stage Docs" directory. The essential idea is consistent coverage.

    4.1 Stage A: 2D Adjacency Prediction (via RFold)

    Title: StageA_2D_Adjacency.md

    Sections:
    1. Purpose & Background
    - Summarize how adjacency prediction is the first step in the pipeline, providing base-pair contacts
    - Reference any relevant paper or logic used (like K-rook approach from RFold)

    1. Inputs & Outputs
    2. Input: Single RNA sequence (string), possible multi-line FASTA, etc.
    3. Output: NxN adjacency matrix (probabilistic or binary)

    4. Possible shapes or thresholding methods described

    5. Key Classes & Scripts

    6. StageARFoldPredictor class
      • Fields: num_hidden, dropout, checkpoint_path, device usage
      • Notable methods: _get_cut_len(), predict_adjacency()

    7. run_stageA.py entry logic: If it downloads missing checkpoints or calls visualization with VARNA

    8. Hydra Configuration

    9. List stageA.num_hidden, stageA.dropout, etc., from stageA.yaml
    10. Provide short code snippet: "In stageA.yaml, we define num_hidden: 128, etc."

    11. Show how StageARFoldPredictor loads these config values

    12. Integration

    13. Upstream: none. (Stage A is the pipeline start for structure)
    14. Downstream: Stage B's TorsionBERT or Pairformer might read adjacency

    15. Additional references: If adjacency is visualized with varna_jar_path

    16. Edge Cases & Logging

    17. If the sequence is < 4 nucleotides, if the checkpoint is missing/corrupted, if a mismatch occurs in dimension, etc.

    18. References

    19. Link to the original RFold paper or relevant GitHub repos
    20. Mention if it depends on a certain PyTorch version or other library for adjacency calculation

    4.2 Stage B: TorsionBERT (LoRA optional) & Pairformer (LoRA optional)

    Title: StageB_TorsionBERT_and_Pairformer.md

    Sections:
    1. Overview & Role
    - This stage outputs local angles (TorsionBERT) and pair embeddings (Pairformer) for each residue/residue pair
    - Quick rationale: angles feed partial 3D builds, pair embeddings capture global context

    1. Inputs & Outputs
    2. Inputs: Possibly adjacency matrix from Stage A, raw sequence, or other features

    3. Outputs: Torsion angles shaped [N, K or 2K] if using sin/cos, single embeddings [N, c_s], pair embeddings [N, N, c_z]

    4. Key Classes

    5. StageBTorsionBertPredictor, PairformerWrapper, plus any support code

    6. If LoRA is used, identify injection points in attention or feed-forward layers

    7. Hydra Config

    8. TorsionBert parameters from stageB_torsion.yaml (like model_name_or_path, lora.enabled, lora.r)
    9. Pairformer parameters from stageB_pairformer.yaml (like c_s, c_z, use_checkpoint)

    10. Show examples of toggling LoRA or specifying target modules in YAML

    11. Integration

    12. Upstream: adjacency or sequence from Stage A

    13. Downstream: Stage C uses angles for partial 3D. Pair embeddings might pass to the Unified Merger

    14. Edge Cases & Error Handling

    15. Sequence indexing issues, mismatch in adjacency dimension, or missing LoRA checkpoint

    16. Potential timeouts or memory constraints if n_blocks or c_z is large

    17. References

    18. If your TorsionBERT is based on an existing HF model or a custom paper, link it
    19. If Pairformer references a particular paper or approach for pairwise embeddings, mention it

    4.3 Stage C: 3D Reconstruction (MP-NeRF or Fallback)

    Title: StageC_3D_Reconstruction.md

    Sections:
    1. Objective & Methods
    - Outline how torsion angles become backbone coordinates, referencing MP-NeRF or the trivial fallback
    - Possibly mention sugar pucker default "C3'-endo" and ring closure logic

    1. Inputs & Outputs
    2. Input: Torsion angles from Stage B, optionally adjacency or sequence

    3. Output: 3D coords shaped [N, #atoms, 3] or [N * #atoms, 3]

    4. Core Classes & Functions

    5. run_stageC_rna_mpnerf(...), StageCReconstruction fallback

    6. Describe how angles are clipped or expanded if user sets a different expected_torsion_count

    7. Hydra Config

    8. stageC.yaml entries: method, do_ring_closure, place_bases, sugar_pucker, etc.

    9. Show how code picks the method string to decide on MP-NeRF or fallback

    10. Integration & Data Flow

    11. Outputs partial 3D coords for the Unified Latent Merger or Stage D directly (depending on your pipeline design)

    12. Mention memory usage, potential HPC constraints for large sequences

    13. Edge Cases & Error Handling

    14. If angles are incomplete or if user tries "legacy" fallback
    15. Negative or out-of-range angles if TorsionBERT is uninitialized, etc.

    Unified Latent Merger

    Title: UnifiedLatentMerger.md

    Sections:
    1. Overview & Importance
    - This merges adjacency, angles, partial coords, single/pair embeddings into a single "conditioning latent" for Stage D
    - Ensures synergy between local angle features and global pair embeddings

    1. Input Structures
    2. For example, [N, dim_angles] for angles, [N, c_s] single embeddings, [N, N, c_z] pair embeddings, [N, #atoms, 3 or N* #atoms, 3] partial coords

    3. Possibly row-wise pooling of pair embeddings to get [N, c_z]

    4. Merger Architecture

    5. If it's a simple MLP (like SimpleLatentMerger), specify hidden layers, activation, dropout

    6. If you might adopt a small transformer in the future, mention that possibility

    7. Hydra Config

    8. From latent_merger.yaml: dim_angles, dim_s, dim_z, dim_out, etc.

    9. How you handle dimension mismatches or expansions (z_pooled = z.mean(dim=1))

    10. Integration

    11. Upstream: results from Stage A/B/C

    12. Downstream: Stage D diffusion uses the merged latent as a conditioning input

    13. Edge Cases

    14. If the shapes from upstream do not match config expectations. Possibly mention logging or shape asserts

    4.4 Stage D: Diffusion-based Refinement & Optional Energy Minimization

    Title: StageD_Diffusion_and_Minimization.md

    Sections:
    1. Purpose
    - Final step refining partial 3D coords or random initialization, possibly using advanced diffusion techniques
    - Optional: short local MD or energy minimization for finishing

    1. Input & Output
    2. Input: Partial coords from Stage C or random noise, plus the unified latent

    3. Output: final 3D structure [N, #atoms, 3], possibly an ensemble from multiple diffusion seeds

    4. Core Modules

    5. run_stageD.py, protenix_diffusion_manager.py, memory optimization logic

    6. Hydra Config

    7. stageD.yaml entries: sigma_data, noise_scale, n_steps, etc.

    8. energy_minimization.enabled, steps, method

    9. Integration

    10. Upstream: Unified latent and possibly partial coords from Stage C

    11. Downstream: Final output, possibly with energy scores or ensemble statistics

    12. Edge Cases & Error Handling

    13. Memory management for large sequences

    14. Handling failed energy minimization or diffusion instability

    15. References

    16. Any papers on RNA-specific diffusion or energy minimization
    17. Links to external tools if used for minimization

    5. Implementation & Maintenance Plan

    1. Initial Documentation Generation
    2. Use automated tools to extract docstrings and signatures

    4. Add code examples and configuration snippets

    5. Review & Validation

    6. Have stage owners review their respective docs
    7. Cross-check with actual code behavior

    8. Verify all config parameters are documented

    9. Integration with Development Flow

    10. Store docs in version control with code
    11. Update docs as part of PR process

    12. Regular review for accuracy

    13. Future Updates

    14. Plan for new features (like Perceiver IO)
    15. Document experimental configurations
    16. Keep HPC notes current

    The goal is maintaining living documentation that stays synchronized with the codebase while providing clear guidance for both new developers and those working on specific pipeline stages.
